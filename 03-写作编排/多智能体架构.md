# 《智能之冬与智能之春》多智能体写作编排架构

> 版本：v1.0 ｜ 2026-07-02
> 本文是本书的**写作方法论文档**，与《智能之冬与智能之春-大纲.md》配套使用。
> 它定义了"如何用多智能体协作工程化地写出这本书"的全部规则。

---

## 一、为什么需要多智能体编排（设计动机）

本书的核心难点不在"写得出来"，而在三个工程约束：

1. **史料零容错**：全书遍布人名、年份、数字（XCON 的 10,000 条规则、第五代机 8.5 亿美元、ChatGPT 5 天破百万）。任何一处硬伤都会摧毁一本技术史的可信度。
2. **结构强对位**：第三章↔第九章是镜像章节，对位表必须逐项精确呼应，写到第九章时不能忘记第三章埋的伏笔。这正是"长程任务漂移"——Agent 自己在第 4000 token 就会忘掉开头。
3. **文风一致性**：10 章约 12,800 字、五个叙事节点，必须维持"史诗感 × 工程严谨 × 批判性反思"的统一基调，不能写散。

**单一 Agent 无法同时满足这三点**——它会幻觉史料、会丢失对位伏笔、会文风漂移。解药是把"写作"拆解为一组**职责单一、可独立验证**的专业角色，每个角色只做一件事、只对一类质量负责。这就是多智能体编排。

这套架构本身就是本书第十章"Agent Operating System"论点的**活体演示**——我们用项目记忆（Project Memory）、状态管理（State Management）、验证层（Verification）来驯服知识的维护成本。

## 二、智能体角色定义（六个角色）

每个角色有明确的**输入、输出、质量责任、禁令**。角色之间只通过结构化产物交接，不直接"对话"。

### 角色 1：Orchestrator（编排者 / 总指挥）

- **职责**：全书的唯一权威调度者。维护全局状态（当前章节、进度、待解决问题清单），决定下一步派谁出场。
- **输入**：大纲 + 各角色的产出 + 待决问题队列。
- **输出**：调度指令（"派 Researcher 核证 X""派 Writer 起草第 N 节"）。
- **质量责任**：全局一致性、对位表完整性、进度。
- **禁令**：不亲自写正文，不亲自核证史料。它只编排，不创作。

### 角色 2：Researcher（史料研究智能体）

- **职责**：针对某一章 / 某个断言，做联网深度检索，收集一手与权威二手来源。
- **输入**：来自 Orchestrator 的研究任务单（"核证 X 事件的人名/年份/数字/原话"）。
- **输出**：**史料卡片包**——每张卡片含：断言、核证状态（✅/⚠️/❌）、确切表述、1–2 个可核实 URL、可信度（高/中/低）。
- **质量责任**：来源可溯、不捏造、明确区分"已核证"与"存疑"。
- **禁令**：绝不生成没有 URL 支撑的"事实"；遇到查不到的，必须标注 ❓ 而非编造。

### 角色 3：Fact-Checker（核证智能体 / 质检关）

- **职责**：Writer 起草后，逐句扫描正文，把每个"事实性断言"抽出来，独立交叉验证（不信任 Researcher 的初判，必须二次复核）。
- **输入**：Writer 的章节草稿。
- **输出**：**核证报告**——每个断言的 ✅/⚠️/❌ 判定 + 修正建议；标注必须改写的句子。
- **质量责任**：是全书史实准确性的最后一道闸门。
- **禁令**：不得为了"读起来顺"而放过存疑断言；任何 ⚠️ 都必须退回 Writer 修改。

### 角色 4：Writer（写作智能体）

- **职责**：基于已核证的史料卡片包，按既定文风规范起草章节正文。
- **输入**：史料卡片包 + 文风规范 + 章节大纲 + 对位伏笔清单。
- **输出**：章节草稿（Markdown）。
- **质量责任**：文风符合"硅谷之火"规范、叙事弧线完整、对位伏笔到位。
- **禁令**：只能使用 Researcher/Fact-Checker 已核证的史料；不得自行"补充"未核证的数字或引语。

### 角色 5：Continuity Editor（连贯性 / 对位编辑）

- **职责**：检查跨章节一致性——第三章埋的伏笔是否在第九章精确呼应、术语是否前后统一、人物是否前后矛盾。
- **输入**：已完成的所有章节 + 全局对位表。
- **输出**：连贯性问题清单（需补 / 需改 / 需删的条目）。
- **质量责任**：本书"思想三段式"（第三章→第九章→第十章）的论证闭环。
- **禁令**：不重写正文，只标注问题交回 Orchestrator。

### 角色 6：Style Editor（文风编辑 / 去 AI 腔）

- **职责**：扫除 AI 腔（排比堆砌、"值得一提的是"之类口水词、空泛赞美、廉价犬儒），强化"具体场景 / 具体数字 / 具体年份"。
- **输入**：章节草稿 + 文风规范。
- **输出**：文风修订建议（逐句标注）。
- **质量责任**：让正文读起来像一个真正在战壕里待过的工程师兼史学家写的，而非模型生成的。
- **禁令**：不得改动事实内容，只调遣词造句。

## 三、写作流水线（Pipeline）

每章按以下流水线推进，**任何一关不通过就退回上一关，绝不跳关**：

```
┌─────────────────────────────────────────────────────────────────┐
│  阶段 0：章节启动                                                 │
│  Orchestrator 读大纲，生成该章"研究任务单"（拆成 N 个核证子任务）  │
└──────────────────────────────┬──────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│  阶段 1：史料研究（可并行）                                        │
│  Researcher × N  并行核证  →  汇总成史料卡片包                      │
│  ◆ 关键：多个 Researcher 同时跑不同子任务，互不干扰                  │
└──────────────────────────────┬──────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│  阶段 2：起草                                                     │
│  Writer 读史料卡片包 + 文风规范 + 对位伏笔清单  →  章节草稿          │
└──────────────────────────────┬──────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│  阶段 3：质检闸门（三关并行扫描同一份草稿）                          │
│  ├─ Fact-Checker：逐句核证史实    →  核证报告                       │
│  ├─ Continuity Editor：查对位伏笔 →  连贯性问题清单                  │
│  └─ Style Editor：查 AI 腔       →  文风修订建议                     │
│  ◆ 任一关发现 ⚠️/❌ → 退回阶段 2 重写，不进入阶段 4                  │
└──────────────────────────────┬──────────────────────────────────┘
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│  阶段 4：定稿                                                     │
│  Writer 按三份报告修订  →  Orchestrator 终审  →  入库（章节库）       │
│  ◆ 同时更新全局对位表与伏笔台账                                     │
└─────────────────────────────────────────────────────────────────┘
```

**并行点**：阶段 1 的多个 Researcher 并行；阶段 3 的三个质检角色并行。串行点是阶段 2→3、3→4——这是**故意串行**的，因为质检必须基于完整草稿，且必须设立闸门。

## 四、全局状态资产（即第十章所说的"工程资产"）

写作不是一次性的，是**跨会话、跨数周**的工程。必须维护五份持久化资产，否则就会重演"上下文坍缩"：

| 资产 | 文件 | 作用 | 状态 |
|---|---|---|---|
| **大纲** | `智能之冬与智能之春-大纲.md` | 全书引力中心，不可擅改 | ✅ v1.0 |
| **写作编排** | `写作编排-多智能体架构.md`（本文档） | 写作"操作系统"，六角色+流水线+文风规范 | ✅ v1.0 |
| **对位台账** | `写作编排-对位台账.md` | 第三章↔第九章每一条对位项的登记/核销（含状态生命周期） | ✅ v1.0 |
| **史料库** | `写作编排-史料库.md` | 全书所有已核证史料的统一仓库，按主题索引，避免重复核证 | ✅ v1.0 |
| **术语表** | `写作编排-术语表.md` | 关键术语中英对照与统一译法 | ✅ v1.0 |

每次写作会话开始，Orchestrator 必须先加载这五份资产作为上下文——这就是第十章所说的 **Project Memory**。

> **资产间的引用契约**：对位台账的每条对位项（A1–A10）都引用史料库的卡片 ID；每章正文只允许引用史料库中 ✅/⚠️ 状态的卡片；术语表为所有资产提供统一词汇。三份资产构成相互校验的闭环——任何一份被擅改，另外两份都能发现不一致。

## 五、文风规范（Style Guide · "硅谷之火"范式）

> 硅谷之火 = 把宏大叙事落到具体的人和具体的瞬间；反对宏大空泛，反对技术报告腔。

### 五条硬性文风指令

1. **每章以一个具体场景 / 人物 / 项目开篇**——禁用"在本章中我们将讨论"。
   - 范式："1987 年，Symbolics 的 3600 机器堆在仓库里卖不动。"
   - 反例："本章将回顾 1987 年 AI 寒冬的成因。"

2. **用具体数字与年份锚定每一个判断**——禁用"许多公司""一段时间后""大量研究"。
   - 范式："ChatGPT 5 天破 100 万用户，2 个月破 1 亿月活。"
   - 反例："ChatGPT 增长很快。"

3. **反对空泛赞美与廉价犬儒**——每个结论必须有结构性解释。
   - 禁止"AI 将彻底改变一切"；禁止"历史总是重演"就交差。
   - 必须回答：为什么是组合爆炸？为什么是维护成本曲线？

4. **AI 腔黑名单**（出现即删）：
   - "值得一提的是""综上所述""众所周知""随着……的发展""在当今……背景下"
   - 三段式排比堆砌（"不仅……而且……更是……"）
   - 形容词通胀（"革命性的""颠覆性的""划时代的"——除非引语）

5. **引语规范**：重要引语中英对照给出，便于读者核查；存疑引语必须标注争议（如 Minsky 1970 Life 那条须注"Minsky 本人否认"）。

### 叙事弧线四段式（每章通用）

- **起**：一个具体场景 / 瞬间 / 人物切入。
- **承**：展开史料，建立事实链条。
- **转**：抛出张力 / 反转 / 阴影。
- **合**：收束于一个钩子，指向下一章。

## 六、多智能体协作的工程约束（防止"Agent 自己重演专家系统病灶"）

这套架构刻意设置了若干约束，以防写作智能体自己掉进本书所批判的坑：

| 约束 | 防范的病灶 | 机制 |
|---|---|---|
| Researcher 禁止生成无 URL 的"事实" | 幻觉 / 记忆污染 | 强制溯源 |
| Fact-Checker 不信任 Researcher 初判，二次复核 | 单点验证失效 | 冗余校验 |
| 任何 ⚠️ 必须退回重写 | Eval 失效导致回退成本爆炸 | 闸门机制 |
| 全局对位台账跨章维护 | 长程任务漂移 / 上下文坍缩 | Project Memory |
| 文风黑名单 + 去 AI 腔扫除 | 风格漂移 | Style Editor |

讽刺且恰当地说：**我们在用第十章的 Agent OS 思想，来写一本论证第十章的书。**

## 七、本章（架构文档）的位置

本文档是写作工程的"操作系统"。每次开始一章前，Orchestrator（由主控智能体扮演）应：
1. 加载本文档 + 大纲 + 四份资产。
2. 按流水线推进。
3. 把每次会话的关键决策记入对位台账。

*（本文档与大纲共同构成本书写作的"双大纲"：内容大纲定写什么，编排大纲定怎么写。）*
