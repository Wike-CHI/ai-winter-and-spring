## 一、台账用途与使用方法

本书的论证骨架是**思想三段式**：

```
第三章（规则帝国·病灶）  →  第九章（历史重演·重演）  →  第十章（Agent OS·解药）
```

第三章和第九章是**严格镜像**：专家系统的每一个结构性病灶，都必须在 Agent 时代找到精确的对应物。第十章则对每一条病灶给出工程解药。

这套对位如果靠记忆维护，写到第九章时必然"长程任务漂移"——这正是本书所批判的 Agent 病灶本身。所以必须用一份**可跨章维护、带状态生命周期的台账**来固化它。

### 状态生命周期

每条对位项有五个状态：

| 状态 | 含义 | 触发条件 |
|---|---|---|
| 🔲 **待埋伏笔** | 尚未写到，第三章对应点待埋 | 默认初始态 |
| 🟡 **已埋伏笔** | 第三章已写入该病灶，待第九章呼应 | 第三章定稿 |
| 🔵 **待呼应** | 第九章开始动笔时标记 | 第九章启动 |
| 🟢 **已呼应** | 第九章已写入 Agent 对应物 | 第九章定稿 |
| ✅ **已核销** | 第十章已写入解药，且 Continuity Editor 确认三章闭环 | 第十章定稿 + 连贯性审查通过 |

### 使用纪律

1. **写第三章时**：每埋一条伏笔，把状态从 🔲 改为 🟡，并在"埋设位置"填入章节小节号。
2. **写第九章时**：开工前把所有 🟡 项扫一遍，确认该呼应哪些；每呼应一条，改为 🟢。
3. **写第十章时**：每给一条解药，改为 ✅。
4. **每章定稿后**：Continuity Editor 必须审查本台账，发现"应呼应未呼应"或"呼应失真"即打回重写。
5. **禁止**：跳过状态直接从 🔲 到 ✅；任何对位项不得"凭印象"标记，必须有正文引用支撑。

## 五、对位审查检查清单（Continuity Editor 用）
